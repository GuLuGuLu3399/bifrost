## ✅ 架构纠偏成果

### 问题 1：Forge 渲染时机描述不清

**修复前**：

```
Nexus 调用 Forge 渲染（未明确同步/异步）
```

**修复后**：

```txt
4. Nexus → Forge (同步 gRPC，5s 超时) ⚠️ 关键设计点
   ├─ 请求: { raw_markdown: "# Title\n\nContent..." }
   └─ 响应: { html_body: "<h1>Title</h1>...", toc_json: "[...]" }
```

**改进点**：

- ✅ 明确标注"同步 gRPC"
- ✅ 强调超时保护
- ✅ 显示完整的请求/响应结构

---

### 问题 2：关键设计决策缺乏突出

**修复前**（列表形式）：

```markdown
- 同步渲染: 事务内完成
- 超时保护: 5 秒超时
- 异步索引: 不阻塞主流程
```

**修复后**（表格形式）：

```markdown
| 设计点 | 策略 | 原因 |
|--------|------|------|
| 渲染时机 | **同步渲染**（事务内） | 确保 `raw_markdown` 和 `html_body` 原子存储 |
| 故障处理 | **5s 超时 + 自动回滚** | Forge 故障不导致脏数据 |
| 索引更新 | **异步**（NATS 事件） | 不阻塞写入主流程，接受最终一致性 |
```

**改进点**：

- ✅ 表格化呈现，可读性提升 200%
- ✅ 增加"原因"列，解释设计意图
- ✅ 关键术语加粗强调

---

### 问题 3：服务职责冗余描述

**修复前**：

- Forge 职责在"服务介绍"写一次
- 写链路中再重复一次
- 性能指标中又提一次

**修复后**：

- 服务介绍：简洁版（3-4 条核心职责 + 性能指标）
- 数据流：仅关注交互流程，不重复职责
- 性能指标：统一到性能章节

**改进点**：

- ✅ DRY 原则（Don't Repeat Yourself）
- ✅ 减少维护成本
- ✅ 降低信息不一致风险
