# Bifrost CMS v3.2 (Pure Edition) - 系统架构蓝图

## 1. 战略规划 (Strategy)

### 核心设计哲学

本架构严格遵循 **CQRS (命令查询职责分离)** 模式，配合 **Go (业务编排)** 与 **Rust (高性能计算)** 的双引擎驱动。

### 代码库策略: Monorepo

我们将所有服务统一管理，确保协议（Proto）与依赖的一致性。

```
bifrost-v3/
├── api/                  # gRPC Protobuf 定义 (通用语言)
│   ├── common/           # 共享消息类型
│   ├── content/          # 内容服务接口
│   │   ├── beacon/       # [Beacon] 读服务接口
│   │   ├── forge/        # [Forge] 渲染服务接口
│   │   ├── nexus/        # [Nexus] 写服务接口
│   │   └── oracle/       # [Oracle] BI 分析接口
│   ├── search/           # [Mirror] 搜索服务接口
│   └── google/           # Google API 扩展
├── go_services/          # Go 服务实现
│   ├── cmd/              # 服务入口
│   │   ├── gjallar/      # [Go] 业务网关 (BFF)
│   │   ├── nexus/        # [Go] 写服务 (Command Side)
│   │   └── beacon/       # [Go] 读服务 (Query Side)
│   ├── internal/         # Go 业务逻辑
│   │   ├── beacon/       # Beacon 业务逻辑
│   │   ├── nexus/        # Nexus 业务逻辑
│   │   ├── gjallar/      # Gjallar 业务逻辑
│   │   └── pkg/          # 公共包
│   ├── configs/          # 服务配置
│   └── api/              # 生成的 Go API 代码
├── rust_services/        # Rust 服务实现
│   ├── forge/            # [Rust] 渲染工作者
│   ├── mirror/           # [Rust] 搜索引擎服务
│   ├── oracle/           # [Rust] BI 数据分析服务
│   └── common/           # Rust 公共库
├── migrations/           # 数据库迁移脚本
├── docker-compose.yml    # 本地开发环境
├── buf.yaml             # Protobuf 管理配置
└── docs/                # 项目文档
```

## 2. 兵力部署 (Service Topology)

### 🛡️ 第一道防线：边缘与网关

#### A. Gjallar (加拉尔) - 咆哮者 (API Gateway / BFF)

- **技术栈**: **Go**
- **职责**:
  - **聚合层 (BFF)**: 面向前端（Web/Mobile）的 API 适配。例如，一个"文章详情页"请求，由 Gjallar 并行调用 Beacon（查内容）和 Mirror（查相关推荐），聚合后返回。
  - **协议转换**: HTTP/JSON <-> gRPC。
  - **短缓存**: 对高频 API 进行秒级微缓存。
  - **认证授权**: JWT 验证、权限控制
  - **限流熔断**: 基于 Token Bucket 的全局限流

### 🧠 核心阵营：Go (业务逻辑 CQRS)

#### B. Nexus (连结者) - 写服务 (Command Side)

- **技术栈**: **Go**
- **职责 (Write Only)**:
  - **全权负责写入**: 用户注册、发布文章、发表评论、点赞。
  - **身份管理 (Identity)**: 处理用户注册、密码修改等写入逻辑。
  - **事务核心**: 维护 `users`, `posts`, `comments` 的主表写入。
  - **发件箱 (Outbox)**: 确保所有写操作都生成事件 (`post.created`, `user.registered`) 推送至 NATS。

#### C. Beacon (灯塔) - 读服务 (Query Side)

- **技术栈**: **Go**
- **职责 (Read Only)**:
  - **高性能读取**: 提供文章列表、详情、用户信息、评论列表的查询。
  - **读模型优化**: 直接读取经过 Forge 渲染好的 `html_body` 和 `toc_json`。
  - **多级缓存**:
    - L1: 本地内存缓存 (BigCache)。
    - L2: 分布式缓存 (Redis)。
  - **视图表**: 维护专门用于列表展示的非规范化数据（如有必要）。

### ⚔️ 力量阵营：Rust (计算与数据)

#### D. Forge (熔炉) - 渲染服务 (Worker)

- **技术栈**: **Rust**
- **职责**:
  - **事件驱动**: 监听 NATS `post.created` / `post.updated`。
  - **Markdown 编译**: 使用 `pulldown-cmark` 将 Nexus 写入的 Raw Markdown 转换为 HTML。
  - **数据清洗**: 防止 XSS 注入。
  - **回写**: 将渲染结果通过 gRPC 回写给 Nexus（或直接更新读库，视一致性要求而定）。

#### E. Mirror (镜像) - 搜索服务 (Search)

- **技术栈**: **Rust**
- **职责**:
  - **全文检索**: 维护倒排索引 (Tantivy / MeiliSearch)。
  - **数据同步**: 消费 NATS 事件，实时构建索引。
  - **搜索接口**: 提供高性能的文章搜索、自动补全、相关性推荐。
  - **替代原静态资源功能**: 之前定义的静态资源代理功能可由 Nginx (Heimdall) 直接承担，Mirror 专注搜索。

#### F. Oracle (神谕) - BI 服务 (Analysis)

- **技术栈**: **Rust**
- **职责**:
  - **数据仓库**: 聚合用户行为数据（浏览、点赞、停留时长）。
  - **离线/实时计算**: 每日生成热度榜单、用户画像分析、内容质量评分。
  - **大屏报表**: 为管理员后台提供数据可视化 API。
  - **定时任务**: 负责清理过期日志、归档旧数据。

## 3. 数据流转 (Data Flow)

### 场景：发布文章 (The Write Path)

1. **User** -> **Gjallar** (校验 JWT) -> **Nexus** (gRPC CreatePost)。
2. **Nexus**:
   - 生成 Snowflake ID。
   - 写入 PostgreSQL `posts` 表 (Status=Draft, Content=Markdown)。
   - 写入 `outbox_events` 表 (Topic=`post.created`)。
   - 返回 "发布成功" 给用户。
3. **Nexus Poller**: 扫描 Outbox，推送消息到 NATS。
4. **Forge (Rust)**: 收到消息 -> 拉取 Markdown -> 渲染 HTML -> 调用 Nexus 回写 HTML -> 标记 Outbox 为 Done。

### 场景：浏览文章 (The Read Path)

1. **User** -> **Gjallar** -> **Beacon** (gRPC GetPost)。
2. **Beacon**:
   - 查 Redis 缓存。
   - (Miss) 查 PostgreSQL `html_body` 字段。
   - 返回 HTML 给用户。

### 场景：搜索文章 (The Search Path)

1. **User** -> **Gjallar** -> **Mirror** (gRPC Search)。
2. **Mirror**:
   - 在 Tantivy 索引中查找。
   - 返回 ID 列表。
3. **Gjallar**: 拿着 ID 列表去 **Beacon** 批量查询文章简要信息 -> 聚合返回。

## 4. 基础设施 (Infrastructure)

- **API Gateway**: Gjallar (Go 实现)
- **Message Queue**: NATS JetStream (持久化、高性能)
- **Database**: PostgreSQL 16 (Nexus/Beacon 共用实例，但逻辑分离；或物理分离)
- **Cache**: Redis 7 Cluster
- **Search**: Tantivy (嵌入式 Rust 库)

## 5. 服务通信

### 5.1 同步通信

- **Gjallar ↔ Beacon**: gRPC (查询操作)
- **Gjallar ↔ Nexus**: gRPC (写入操作)
- **Forge ↔ Nexus**: gRPC (回写渲染结果)
- **Gjallar ↔ Mirror**: gRPC (搜索操作)

### 5.2 异步通信

- **Nexus → Forge**: NATS (文章创建/更新事件)
- **Nexus → Beacon**: NATS (数据变更通知)
- **Nexus → Oracle**: NATS (用户行为事件)

## 6. 部署架构

### 6.1 容器化

### 6.2 服务发现

- **开发环境**: Docker Compose 网络内直接通信
- **生产环境**: Kubernetes Service + DNS

### 6.3 配置管理

- **Go 服务**: YAML 配置文件 + 环境变量覆盖
- **Rust 服务**: TOML 配置文件 + 环境变量覆盖

## 8. 安全设计

### 8.1 认证授权

- **JWT Token**: 无状态认证
- **RBAC**: 基于角色的访问控制
- **Token 刷新**: 自动刷新机制

### 8.2 数据安全

- **传输加密**: TLS 1.3
- **存储加密**: 敏感数据加密存储
- **SQL 注入防护**: 参数化查询
- **XSS 防护**: 输入验证和输出编码

### 8.3 网络安全

- **API 限流**: 防止 DDoS
- **CORS 策略**: 跨域请求控制
- **WAF**: Web 应用防火墙

## 9. 性能优化

### 9.1 缓存策略

- **多级缓存**: 内存缓存 + 分布式缓存
- **缓存预热**: 热点数据预加载
- **缓存更新**: 主动失效 + 定时刷新

### 9.2 数据库优化

- **读写分离**: CQRS 模式天然支持
- **连接池**: 优化数据库连接管理
- **索引优化**: 基于查询模式设计索引

### 9.3 并发处理

- **Go**: Goroutine 池管理
- **Rust**: Tokio 异步运行时
- **消息队列**: 异步解耦处理

## 12. 开发流程

### 12.1 本地开发

```bash
# 启动基础服务
docker-compose up -d postgres redis nats

# 启动 Go 服务
cd go_services
go run cmd/nexus/main.go -f configs/nexus.yaml
go run cmd/beacon/main.go -f configs/beacon.yaml
go run cmd/gjallar/main.go -f configs/gjallar.yaml

# 启动 Rust 服务
cd rust_services
cargo run --bin forge
cargo run --bin mirror
cargo run --bin oracle
```

### 12.2 API 定义

所有 API 接口通过 Protobuf 定义，位于 `api/` 目录：

```protobuf
// api/content/v1/nexus/nexus.proto
service NexusService {
  rpc CreatePost(CreatePostRequest) returns (CreatePostResponse);
  rpc UpdatePost(UpdatePostRequest) returns (UpdatePostResponse);
  rpc DeletePost(DeletePostRequest) returns (DeletePostResponse);
}
```

### 12.3 代码生成

```bash
# 生成 Go 代码
buf generate

# 生成 Rust 代码
cd rust_services
cargo build
```
